# 第1章 ロードマップ ワークブック

このワークブックは、第1章の演習「研修ロードマップを自分の言葉で説明する」で使う。

## 使い方

この資料を読んでから、`assignments/chapter01-roadmap/submission/<github-account>/chapter01-submission.md` を書く。目的は、24章の名前を覚えることではなく、「自分は何を、どの順番で、なぜ学ぶのか」を説明できるようにすることである。

## 前提

`<github-account>` は自分のGitHubアカウント、または講師が指定した識別子に置き換える。記入用ファイルは `starter/chapter01-submission.md` を使う。

## 研修全体の構成

この研修は6つのPartで構成する。

| パート | 何を扱うか | できるようになってほしいこと |
| --- | --- | --- |
| パート1: 仕事と学び方 | 研修の目的、事業価値、AI利用、相談とレビュー | 何のために学ぶのかを説明し、分からないことを前に進められる |
| パート2: 手元で動かす基礎 | 開発環境、Git/GitHub、Webの仕組み、既存コード読解 | 手元で動かし、変更し、レビューに出せる |
| パート3: Webアプリを設計して作る | 要件、DB、API、フロントエンド、テスト | 小さなWebアプリの中心部分を設計、実装、確認できる |
| パート4: 安全に届けて運用する | セキュリティ、Docker、AWS、CI/CD、オブザーバビリティ | 作ったものを安全に公開し、問題を調査できる |
| パート5: AIネイティブエンジニアリング | LLM、Claude Code、テクニカルライティング | AIを使いながら、出力を検証し、判断を文書化できる |
| パート6: 統合プロジェクト | 個人開発、既存プロダクト改善、最終確認、発表 | 自由テーマの個人プロダクトを完成させ、説明できる |

## 配属直後の小さなタスクで考える

例:

> ログイン画面に補足文言を追加してほしい。ついでに、入力エラー時の表示が分かりにくいので直してほしい。

このタスクを進めるには、次のような力が必要になる。

| 作業 | 関係するパート |
| --- | --- |
| 何を直すべきかを確認する | パート1, パート3 |
| ローカルで画面を動かす | パート2 |
| 既存コードからログイン画面を探す | パート2 |
| エラー表示のHTMLや画面状態を直す | パート3 |
| テストや確認手順を書く | パート3 |
| セキュリティやアクセシビリティの観点で確認する | パート4 |
| Claude Codeに相談し、差分を自分で確認する | パート5 |
| PRで変更理由を説明する | パート1, パート5 |

## 個人開発課題で考える

個人開発課題は自由テーマだが、評価される構造は共通である。

| 個人開発課題に必要な要素 | 関係するパート |
| --- | --- |
| 誰のどんな課題を解くかを決める | パート1, パート3 |
| 画面とデータを設計する | パート3 |
| APIやDBを実装する | パート3 |
| 手元で動かし、GitHubで管理する | パート2 |
| テストや確認手順を用意する | パート3 |
| AWSにデプロイする | パート4 |
| セキュリティ、アクセシビリティ、運用観点を確認する | パート4 |
| Claude Codeの利用箇所と検証結果を残す | パート5 |
| 最後に成果と判断理由を発表する | パート6 |

## 考えるための問い

提出ファイルでは、次の問いに答える。

1. 6つのPartのうち、今の自分にとって一番重要そうなPartはどれか。
2. それを選んだ理由は何か。自分の経験、不安、作りたいものと接続して書く。
3. そのPartを学ぶと、個人開発課題のどの作業が進めやすくなるか。
4. 逆に、そのPartを理解しないままだと、どこで困りそうか。
5. 研修中にメンターへ確認したいことは何か。

## 記入例

```md
## 自分にとって重要そうなパート

パート2: 手元で動かす基礎

## 理由

個人開発で画面を作ったことはあるが、エラーが出たときにログや設定を見て原因を探すのが苦手だから。GitHubも個人利用が中心で、PRを使った開発はまだ慣れていない。

## 個人開発課題との接続

自由テーマでアプリを作るとき、まずローカルで安定して動かせないと、DB、API、画面、テストのどこで問題が起きているか分からなくなる。パート2でローカル実行、GitHub、Webの基本、既存コード読解を押さえると、後半の開発が進めやすくなる。

## 分からないままだと困りそうなこと

エラーが出たときに、AIに聞くだけで終わり、自分で再現条件や確認手順を説明できない可能性がある。

## メンターに確認したいこと

エラー調査のとき、最初にログ、設定、コードのどこを見るべきかの順番を知りたい。
```
