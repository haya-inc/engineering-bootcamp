# 第23章 本番準備 ワークブック

このワークブックは、第23章の演習「リリース候補を整理する」「本番準備チェックリストを作る」「セキュリティとアクセシビリティを確認する」「パフォーマンスとオブザーバビリティを確認する」「手順書、スモークテスト、リリース判断を書く」で使う。

## 使い方

この資料と、第21章、第22章で作成した成果物を読み、`release-candidate-summary.md`、`production-readiness-checklist.md`、`security-accessibility-readiness.md`、`performance-observability-readiness.md`、`release-runbook.md`、`smoke-test-plan.md`、`release-decision.md`、`follow-up-issues.md` を作る。

目的は、正式監査を行うことではない。リリースしてよいかを判断するために、確認したこと、確認できていないこと、残すリスクを説明できる状態にすることである。

## 前提

第22章で作った改善PRをrelease candidateとして扱う。

共通例:

```txt
学習ログ整理アプリにstatusを追加し、needs-helpだけを絞り込めるようにする。
```

研修でAWS環境を使う場合は、Amazon ECS（Fargate 起動タイプ）、ECR、RDS、Secrets ManagerまたはParameter Store、CloudWatch Logsを標準構成として想定する。
新しい環境で使う場合は、使う時点のAWS公式ドキュメントで、ECS（Fargate）の設定（cluster・task definition・service、task execution role、awsvpcネットワーク）を確認する。
AWS環境を使わない場合も、AWSでリリースするならどこを確認するかを説明する。

扱わないこと:

- 正式なセキュリティ監査
- ペネトレーションテスト
- 大規模負荷試験
- 24時間オンコール設計
- 組織横断の承認プロセス

## 演習1: リリース候補を整理する

### 考えること

リリース前確認では、まず確認対象を固定する。何をリリースするのかが曖昧だと、確認観点も曖昧になる。

問い:

- 今回リリースする変更は何か。
- 今回リリースしないものは何か。
- 主要ユーザーフローは何か。
- 既知の問題は何か。
- どの環境で確認するか。
- どのPR、commit、image tag、artifactを確認対象にするか。
- 判断に使う証拠はどこにあるか。

### 記録すること

`release-candidate-summary.md` は次の形で書く。

```md
# リリース候補サマリー

## プロダクト


## リリース候補


## 含める変更

- 

## 含めないもの

-

| item | value | エビデンス |
| --- | --- | --- |
| branch or PR |    |    |
| commit SHA or image tag |    |    |
| migration note version |    |    |
| release note version |    |    |

## 主要ユーザーフロー

| フロー | 重要な理由 | エビデンス |
| --- | --- | --- |
|    |    |    |

## 証拠一覧

| evidence | status | link or note |
| --- | --- | --- |
| regression test plan | 合格 / 不合格 / 一部 |    |
```

### チェック

- [ ] 何をリリースするかが明確である

- [ ] 主要ユーザーフローが3つ以内に絞られている
- [ ] 既知の問題が書かれている
- [ ] 確認環境が書かれている

## 演習2: 本番準備チェックリストを作る

### 考えること

チェックリストは、確認を機械的に終わらせるためではなく、見落としを減らすために使う。

問い:

- securityで何を見るか。
- accessibilityで何を見るか。
- performanceで何を見るか。
- observabilityで何を見るか。
- operationsで何を見るか。
- documentationで何を見るか。
- 確認できない項目は何か。

### 記録すること

`production-readiness-checklist.md` は次の形で書く。

## 演習3: セキュリティとアクセシビリティを確認する

### 考えること

リリース前に、利用者を危険にしたり、使えない状態にしたりしないかを見る。

問い:

- 不正な入力を保存できないか。
- 権限外の操作を許していないか。
- 秘密情報、個人情報、本番データがログ、Git、AI入力、READMEに混ざっていないか。
- AWSアカウントID、内部URL、実ログのrequest idなど、公開提出やAI入力に不要な環境識別情報を含めていないか。
- 依存関係に明らかな問題はないか。
- keyboardだけで主要操作ができるか。
- label、focus、error messageは分かるか。
- 色だけで状態を伝えていないか。

### 記録すること

`security-accessibility-readiness.md` は次の形で書く。

```md
# セキュリティとアクセシビリティ準備確認

## セキュリティ確認

| 確認 | 方法 | 期待結果 | result | エビデンス | リリース影響 |
| --- | --- | --- | --- | --- | --- |
| 入力検証 |    | 不正な値を保存しない | 合格 / 不合格 / 省略 |    |    |
| 認可 |    | 権限外操作を拒否する | 合格 / 不合格 / 省略 |    |    |
| 秘密情報の扱い |    | secret、個人情報、本番データを含めない | 合格 / 不合格 / 省略 |    |    |
| 依存関係確認 |    | high/criticalがあれば説明する | 合格 / 不合格 / 省略 |    |    |
| ログ安全性 |    | secret、個人情報、不要な環境識別情報を出さない | 合格 / 不合格 / 省略 |    |    |

## アクセシビリティ確認

| 確認 | 方法 | 期待結果 | result | エビデンス | リリース影響 |
| --- | --- | --- | --- | --- | --- |
| キーボード操作 |    | 主要操作を完了できる | 合格 / 不合格 / 省略 |    |    |
| ラベル |    | 入力や選択の意味が分かる | 合格 / 不合格 / 省略 |    |    |
| フォーカス表示 |    | 現在位置が見える | 合格 / 不合格 / 省略 |    |    |
| エラーメッセージ |    | 何を直せばよいか分かる | 合格 / 不合格 / 省略 |    |    |
| 色だけに依存しない |    | 文字や形でも状態が分かる | 合格 / 不合格 / 省略 |    |    |

## ブロッカー

- 不正入力を保存できる:
- 権限外データが見える:
- secretや実在データが露出している:
- 主要操作がキーボードでできない:

## フォローアップ

| 課題 | 理由 | 担当 | 期限 |
| --- | --- | --- | --- |
|    |    |    |    |
```

### チェック

- [ ] securityの結果がある
- [ ] accessibilityの結果がある
- [ ] 証拠がある
- [ ] blockerとfollow-upが分かれている
- [ ] secretや個人情報を含めていない
- [ ] AWSアカウントIDや内部URLなど、不要な環境識別情報を含めていない

## 演習4: パフォーマンスとオブザーバビリティを確認する

### 考えること

主要操作が現実的に使え、問題が起きたときに調査できるかを見る。

問い:

- 主要画面は重すぎないか。
- 一覧件数が増えた場合に問題になりそうか。
- APIレスポンスは大きすぎないか。
- エラーが起きたとき、どのログを見るか。
- CloudWatch Logsなどで確認する場所が分かるか。
- ログにsecretや個人情報が出ていないか。

### 記録すること

`performance-observability-readiness.md` は次の形で書く。

## AIに相談するときの例

AIに渡してよい情報だけを使い、秘密情報、実在の顧客情報、実在の個人情報、本番データ、AWSアカウントID、内部URLは含めない。

```txt
第22章で作った改善PRについて、研修用の本番準備レビューを行います。

リリース候補:
学習ログにstatusを追加し、needs-helpだけを絞り込めるようにする。

確認済み:
- 既存ログは表示される
- 新規ログにstatusを保存できる
- 不正なstatusは保存されない
- READMEは更新済み

未確認:
- 一覧件数が増えた場合の性能
- CloudWatch Logsでのエラー確認

お願い:
1. security、accessibility、performance、observability、operations、documentationの観点で抜け漏れを挙げてください。
2. go / go with follow-up / no-go判断に必要な追加確認を提案してください。
3. スモークテストを3項目に絞って提案してください。
4. release decision文書の不足をレビューしてください。
5. 提案を、採用するもの、省略して理由を書くもの、follow-upにするものへ分けてください。
```

AIの提案は、実行結果、コード差分、ログ、公式資料で確認する。AIの判断だけでgoにしない。
