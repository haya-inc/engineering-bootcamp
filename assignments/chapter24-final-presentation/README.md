# 第24章 最終発表 ワークブック

このワークブックは、第24章の演習「最終エビデンス一覧を作る」「最終発表ブリーフを書く」「最終デモ台本を作る」「学習の振り返りを書く」「次の90日計画を書く」で使う。

## 使い方

第1章から第23章までに作った成果物を読み、`final-evidence-index.md`、`final-presentation-brief.md`、`final-demo-script.md`、`learning-reflection.md`、`next-90-days-plan.md` を作る。

目的は、発表の見栄えを整えることではない。何を作り、なぜ作り、どう確認し、何を学び、次に何を伸ばすかを説明できる状態にすることである。

## 前提

最終発表では、次を説明する。

- ユーザー、課題、成果
- 作ったもの
- 重要な技術判断
- デモ
- 品質確認とPRR結果
- AI利用と検証
- 失敗、迷い、削ったこと
- 残課題
- 次の90日計画

発表の上手さだけを目的にしない。発表資料と実際の成果物、確認ログ、PRR結果が一致していることを重視する。

## 演習1: 最終エビデンス一覧を作る

### 考えること

最終発表で使う証拠を選ぶ。すべての成果物を見せる必要はない。

問い:

- 何を作ったことの証拠はどれか。
- なぜ作ったことの証拠はどれか。
- どう確認したことの証拠はどれか。
- AIをどう使ったことの証拠はどれか。
- 次に何を改善するかの証拠はどれか。
- 発表で主張する内容に、対応する証拠はあるか。
- 見せてはいけない情報を、安全な代替証拠に置き換えられるか。

### 記録すること

`final-evidence-index.md` は次の形で書く。

```md
# 最終エビデンス一覧

## 中心となるエビデンス

| エビデンス | 支える主張 | source of truth | 発表での使い方 | 見せてよいか | 代替証拠 |
| --- | --- | --- | --- | --- | --- |
| project-brief.md |    |    |    | yes / redact / no |    |
| mvp-scope.md |    |    |    | yes / redact / no |    |
| improvement-pr-summary.md |    |    |    | yes / redact / no |    |
| production-readiness-checklist.md |    |    |    | yes / redact / no |    |
| release-decision.md |    |    |    | yes / redact / no |    |

## 補足エビデンス

| エビデンス | 支える主張 | いつ使うか | 見せてよいか |
| --- | --- | --- | --- |
|    |    |    | yes / redact / no |

## 不足しているエビデンス

| 不足項目 | 影響する主張 | 発表での扱い | フォローアップ |
| --- | --- | --- | --- |
|    |    |    |    |

## 見せないもの

| 項目 | 理由 | 安全な代替 |
| --- | --- | --- |
|    |    |    |
```

### チェック

- [ ] 発表で使う証拠が選ばれている
- [ ] 証拠の役割が書かれている
- [ ] 主張と証拠が対応している
- [ ] 見せないものも理由つきで分けている
- [ ] 見せられない情報に安全な代替証拠がある
- [ ] 秘密情報、実在の顧客情報、実在の個人情報、本番データ、AWSアカウントID、内部URL、実ログ全文を含めていない

## 演習2: 最終発表ブリーフを書く

### 考えること

発表の中心メッセージを決める。できたことの羅列ではなく、価値、判断、確認、学びの流れにする。

問い:

- 1文で言うと、何をした研修成果か。
- 誰のどんな課題を扱ったか。
- どの技術判断が重要だったか。
- どの品質確認を行ったか。
- 何が残課題か。
- 強い主張に証拠があるか。
- 証拠がない主張を、未確認またはfollow-upとして弱められているか。

### 記録すること

`final-presentation-brief.md` は次の形で書く。

```md

## 核となるメッセージ


## 想定読者


## ユーザー、課題、成果

- user:
- problem:
- outcome:

## プロダクト概要


## 主要な技術判断

| 判断 | 理由 | トレードオフ | エビデンス |
| --- | --- | --- | --- |
|    |    |    |    |

## 品質エビデンス

| 領域 | エビデンス | result |
| --- | --- | --- |
| テスト |    |    |
| セキュリティ |    |    |
| アクセシビリティ |    |    |
| 運用 |    |    |
| 文書化 |    |    |

## AI利用

| 段階 | AIの役割 | 人間の確認 | 採用したか | エビデンス |

1.
```

### チェック

- [ ] 中心メッセージが1文で書かれている
- [ ] 価値、技術判断、品質確認がつながっている
- [ ] AI利用と検証が書かれている
- [ ] 採用しなかったAI提案も必要に応じて書かれている
- [ ] 証拠がない主張を強く言いすぎていない
- [ ] フィードバックしてほしい観点がある

## 演習3: 最終デモ台本を作る

### 考えること

デモは、機能一覧を順番に見せる時間ではない。主要ユーザーフローを使って、価値と動作を見せる。

問い:

- 最初にどんな状態を見せるか。
- どの操作で価値が伝わるか。
- 期待結果は何か。
- デモが失敗したら何を見せるか。
- デモ後に何の確認結果を補足するか。
- 画面に見せてはいけない情報が出そうな場合、どう止めるか。

### 記録すること

`final-demo-script.md` は次の形で書く。

```md
# 最終デモ台本

## デモの目的


## デモの安全確認と準備

- 使用するデータ:
- 見せないもの:
- 起動コマンド:
- 確認済みcommitまたはartifact:

## 初期状態


## フロー1

| 手順 | 対応 | 期待結果 | メモ |
| --- | --- | --- | --- |
| 1 |    |    |    |
| 2 |    |    |    |
| 3 |    |    |    |

## フロー2

| 手順 | 対応 | 期待結果 | メモ |
| --- | --- | --- | --- |
| 1 |    |    |    |
| 2 |    |    |    |

## デモが失敗した場合

| 問題 | その場で話すこと | 代替エビデンス |
| --- | --- | --- |
|    |    |    |

## デモ後

- quality evidence:
- known limitation:
- next improvement:
```

### チェック

- [ ] デモの目的がある
- [ ] 主要フローが1つから2つに絞られている
- [ ] 期待結果が書かれている
- [ ] 失敗時の代替証拠がある
- [ ] 見せてはいけない情報を出さない準備がある
- [ ] デモ後に補足する品質確認がある

## 演習4: 学習の振り返りを書く

### 考えること

うまくいかなかったことや削ったことを、次の行動に変換する。

問い:

- うまくいったことは何か。
- 詰まったことは何か。
- 判断を変えたことは何か。
- AI利用で学んだことは何か。
- 次に同じ仕事をするとき、何を変えるか。

`learning-reflection.md` は次の形で書く。

```md
|    |    |    |

|    |    |    |

| 削った項目 | 理由 | 今後の選択肢 |

- verification:
```

- [ ] AI利用の学びがある
- [ ] 次に変える行動がある

## 演習5: 次の90日計画を書く

### 考えること

次の90日計画は、抽象目標ではなく、配属後の行動と成果物にする。

問い:

- 次に伸ばす領域は何か。
- 30日後、60日後、90日後に何が変わっているとよいか。
- どんな成果物を出すか。
- 成果物ができたと言えるsuccess signalは何か。
- 誰に何を相談するか。
- 最終発表のフィードバックをどう反映するか。
- 週次で何を見直すか。

### 記録すること

`next-90-days-plan.md` は次の形で書く。

```md
# 次の90日計画

## 注力領域

1. 
2.
```

## AIに相談するときの例

AIに渡してよい情報だけを使い、秘密情報、実在の顧客情報、実在の個人情報、本番データは含めない。
AWSアカウントID、内部URL、実ログ全文も、不要なら含めない。

```txt
最終発表の構成を作っています。

中心メッセージ:
研修中の学習ログを整理し、相談が必要なログを見つけやすくするWebアプリを作り、status追加の改善とリリース前確認まで行いました。

発表に入れたい内容:
- ユーザー、課題、成果
- デモ
- 重要な技術判断
- 品質確認とPRR結果
- AI利用と検証
- 失敗と学び
- 次の90日計画

お願い:
1. 発表構成をレビューしてください。
2. 成果物と説明がつながっていない箇所を指摘してください。
3. 技術判断の説明として不足しそうな観点を挙げてください。
4. next 90 days planを、行動と成果物が分かる形に改善してください。
5. 私の成果物に根拠がない話が混ざっていないか指摘してください。
```

AIの出力をそのまま発表内容にしない。自分の成果物、確認結果、フィードバックに合わせて修正する。
