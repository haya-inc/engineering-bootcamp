# 第6章 GitとGitHub ワークブック

このワークブックは、第6章の演習「リポジトリの現在地を確認する」「小さな変更をブランチとコミットにする」「プルリクエストを作る」「レビューコメントに対応する」「競合と履歴確認の練習をする」で使う。

## 使い方

この資料を読み、`git-work-log.md`、`pull-request-note.md`、`review-response-log.md`、`conflict-and-history-note.md` を書く。

目的は、Gitコマンドを暗記することではない。自分がどのrepositoryで、どのbranchにいて、何を変更し、どの差分をレビューに出したのかを説明できるようにすることである。

## GitとGitHubを見分ける

この章では、GitとGitHubを分けて考える。

| 観点 | Git | GitHub |
| --- | --- | --- |
| 何をするものか | 手元で変更履歴を作る | 変更履歴を共有し、PRやレビューを行う |
| どこで見るか | ターミナル、エディタ | ブラウザ、GitHub CLI |
| このワークブックで記録すること | ブランチ、diff、コミット | プルリクエストURL、PR本文、レビューコメント |

迷ったら、`git status` や `git diff` で見るものはGit、プルリクエストやレビューコメントで見るものはGitHub、と考える。

## 前提

演習では、このリポジトリの `starter-apps/learning-log-sample` を使う。プロジェクト名は `learning-log-sample` である。

題材:

> メンター向け進捗一覧で、担当受講者が0件のときに空白画面に見えてしまう。空状態の案内文を追加したい。

第4章の `change-description.md` と、第5章の `local-run-log.md` を材料として使う。

## 安全ルール

研修中は、意味が分からないまま次の操作をしない。

- `git reset --hard`
- `git clean -fd`
- `git push --force`
- `git rebase`
- `git cherry-pick`

必要になった場合は、メンターに相談する。AIが提案した場合も、そのまま実行しない。

## 演習1: リポジトリの現在地を確認する

### 実行すること

演習用repositoryのルートに移動し、現在地を確認する。

```txt
pwd
git status
git branch --show-current
git remote -v
```

必要に応じて履歴を確認する。

```txt
git log --oneline -5
```

### 記録すること

`git-work-log.md` は次の形で書く。

```md
# Git作業ログ

## 作業リポジトリ


## 作業前のブランチ


## 作業前の状態


## リモート


## 最新のコミット


## 気づいたこと、確認したいこと

- 
```

## 演習2: 小さな変更をブランチとコミットにする

### 実行すること

作業用branchを作る。

```txt
git switch -c feature/progress-empty-state
```

サンプルアプリに、小さな変更を入れる。

変更例:

- 進捗一覧が空のときに案内文を表示する。
- 担当受講者が0件の場合の表示だけを変更する。
- 既存の一覧表示は変えない。

変更後、差分を確認する。

```txt
git status
git diff
```

意図した変更だけであれば、コミットする。

```txt
git add <changed-files>
git commit -m "Add empty state copy for progress list"
```

### コミット前チェック

- [ ] 変更したファイルは意図どおりである
- [ ] 関係ない変更が混ざっていない
- [ ] APIキー、トークン、個人情報が含まれていない
- [ ] ローカルで最低限の確認をした
- [ ] コミットメッセージが変更意図を表している

### 記録すること

`git-work-log.md` に追記する。

```md
## 作業ブランチ


## 変更内容

- 

## 差分確認で見たこと

- 

## コミットメッセージ


## ローカル確認

- 
```

## 演習3: プルリクエストを作る

### 実行すること

branchをGitHubへpushする。

```txt
git push -u origin feature/progress-empty-state
```

GitHub上でプルリクエストを作る。

第4章の `change-description.md` をもとに、PR本文を書く。第5章の `local-run-log.md` から、確認方法も転記する。

### PR本文の型

`pull-request-note.md` は次の形で書く。

```md
# プルリクエストメモ

## PR URL


## 背景


## 変更内容

- 

## 今回やらないこと

- 

## 確認方法

- 

## 影響範囲、リスク


## レビューしてほしい点

- 

## AIを使った場合の検証

```

### 悪い例

```md
進捗一覧を直しました。
```

足りない情報:

- なぜ直したか。
- 何を変えたか。
- 何を変えていないか。
- どう確認したか。
- レビュアーに何を見てほしいか。

### 良い例

```md
## 背景

担当受講者がまだいないメンターが進捗一覧を開くと、空白画面に見えてしまいます。

## 変更内容

- 進捗一覧が空の場合に案内文を表示しました。
- 次に行うべき操作として「担当受講者が割り当てられると表示されます」と示しました。

## 今回やらないこと

- 未提出者フィルタの仕様変更
- 自動アラート

## 確認方法

- 担当受講者が0件の状態で進捗一覧を開く。
- 案内文が表示されることを確認する。
- 担当受講者がいる状態では、既存の一覧表示が変わらないことを確認する。

## レビューしてほしい点

- 案内文が初回リリースの表現として自然か。
- 既存の一覧表示に不要な影響がないか。
```

## 演習4: レビューコメントに対応する

次のレビューコメントに対応する。

### コメントA

```txt
担当受講者が0件の場合と、フィルタ結果が0件の場合で表示文言を分けたほうがよさそうです。
```

対応で書くこと:

- 指摘をどう理解したか。
- 修正するか、質問するか。
- 修正する場合、どの表示を分けるか。
- 追加でどう確認するか。

### コメントB

```txt
この文言は少し断定的に見えます。メンターが次に何をすればいいかが分かる表現にできますか？
```

対応で書くこと:

- 文言の目的をどう理解したか。
- 代替文言案。
- レビューしてほしい点。

### コメントC

```txt
今回のPRで未提出者フィルタの挙動も直せますか？
```

対応で書くこと:

- 今回のPRの目的との関係。
- このPRに含めるか、別PRに分けるか。
- 分ける場合の理由。

### 記録すること

`review-response-log.md` は次の形で書く。

```md
# レビュー対応ログ

## コメントA

### 判断

修正する / 質問する / 見送る

### 理由


### 返信案


### 追加コミット


## コメントB

### 判断


### 理由


### 返信案


### 追加コミット


## コメントC

### 判断


### 理由


### 返信案


### 追加コミット

```

## 演習5: 競合と履歴確認の練習をする

### 競合例

次のconflict markerを読む。

```txt
<<<<<<< HEAD
担当受講者が割り当てられると、ここに進捗が表示されます。
=======
条件に一致する受講者はいません。フィルタ条件を変更してください。
>>>>>>> feature/filter-empty-state
```

考えること:

- 上側の変更は何を意図しているか。
- 下側の変更は何を意図しているか。
- 2つの状態は同じか、別物か。
- 最終的にどの文言を残すべきか。
- 判断に迷う場合、誰に何を確認するか。

### 履歴確認

履歴を見るコマンド例:

```txt
git log --oneline -5
git show <commit-sha>
```

GitHub上でも、PRの会話、レビューコメント、コミット一覧を確認する。

### 記録すること

`conflict-and-history-note.md` は次の形で書く。

```md
# 競合と履歴確認メモ

## 競合で起きていること


## 上側の変更意図


## 下側の変更意図


## 最終的に残す内容


## その理由


## 履歴から確認したこと


## 相談する場合に伝えること

- 
```

## AIを使う場合

AIに頼んでよいこと:

- `git diff` の要約
- PR本文の抜け漏れ確認
- コミットメッセージ案
- レビューコメントへの返信案
- conflict markerの読み方の説明

AIに任せたままにしてはいけないこと:

- 差分を読まずにコミットする
- 実行していない確認をPR本文に書く
- 意味が分からないGitコマンドを実行する
- 破壊的なコマンドを独断で実行する

AIへの依頼例:

```txt
以下は研修用PRの差分要約とPR本文の下書きです。

目的:
レビューしやすいPR本文にしたいです。

前提:
- 実際の差分は自分で確認済みです。
- 秘密情報は含めていません。

出してほしいこと:
- PR本文に足りない観点
- 確認方法の抜け漏れ
- レビュアーに見てほしい点の改善案

注意:
- 実行していない確認を追加しないでください。
- 不確かなことは断定しないでください。
```

- [ ] プルリクエストURLを記録した

- [ ] `review-response-log.md` に修正、質問、見送りの判断がある
- [ ] `conflict-and-history-note.md` に競合例の判断理由がある
- [ ] 秘密情報、トークン、個人情報をコミットやPR本文に含めていない
- [ ] AIを使った場合、出力を自分で確認した
