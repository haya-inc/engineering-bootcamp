# 第8章 デバッグ ワークブック

このワークブックは、第8章の演習「不具合を再現する」「関連コードを探す」「原因候補を仮説として書く」「最小修正と確認を行う」「修正PRを作る」で使う。

## 使い方

この資料を読み、`reproduction-log.md`、`code-reading-note.md`、`hypothesis-log.md`、`fix-verification-log.md`、`bug-fix-pr-note.md` を書く。

目的は、すぐに正解の修正を当てることではない。再現、観察、仮説、確認、修正説明を順番に残し、あとから自分とレビュアーが判断を追えるようにすることである。

## 前提

このリポジトリの `starter-apps/learning-log-sample` を使う。プロジェクト名は `learning-log-sample` である。

第5章から第7章の成果物を使う。

- `project-overview.md`
- `local-run-log.md`
- `pull-request-note.md`
- `http-observation-log.md`

## 題材

メンター向け進捗一覧に、次の不具合がある。

```txt
未提出者フィルタを選んでも、提出済みの受講者が一覧に残る。
```

この不具合は `starter-apps/learning-log-sample` に演習用として含まれている。

期待結果:

- `submittedAt` が `null` の受講者だけが表示される。

実際の結果:

- `submittedAt` に日時が入っている受講者も表示される。

## 安全ルール

デバッグ中は、次のことを守る。

- 再現する前に修正を始めない。
- 観察結果と推測を混ぜない。
- 不具合修正とリファクタリングを混ぜすぎない。
- 秘密情報、Cookie、token、個人情報をログやAI入力に含めない。
- AIが出した修正を、差分確認なしにcommitしない。

## 演習4: 最小修正と確認を行う

### 実行すること

1. 第6章の手順で作業branchを作る。
2. 残った仮説に対して、最小修正を入れる。
3. `git diff` で差分を見る。
4. 再現手順をもう一度実行する。
5. 関連する正常系も確認する。
6. 既存テストがあれば実行する。

branch名の例:

```txt
fix/unsubmitted-progress-filter
```

確認観点:

- 未提出者フィルタで未提出者だけが表示される。
- フィルタなしでは全件が表示される。
- 提出済みフィルタがある場合、提出済みだけが表示される。
- APIレスポンスが期待どおりである。
- 既存テストが通る。

## 演習5: 修正PRを作る

### 実行すること

1. `git status` と `git diff` で差分を確認する。
2. コミットする。
3. branchをGitHubへpushする。
4. プルリクエストを作る。
5. 再現、原因、修正内容、確認方法、リスクを書く。

### PR本文の材料

第8章の成果物を使う。

- `reproduction-log.md`: 再現手順、期待結果、実際の結果
- `code-reading-note.md`: 変更箇所、影響範囲
- `hypothesis-log.md`: 原因候補、確認結果
- `fix-verification-log.md`: 修正後の確認

## AIを使う場合

AIに頼んでよいこと:

- 原因候補の洗い出し
- 検索すべき文字列の提案
- スタックトレースの読み方の説明
- 修正案の比較
- テスト観点の候補出し
- PR本文の抜け漏れ確認

AIに最初から任せすぎないこと:

- 再現前に修正コードを書かせる
- 既存コード全体を大きく書き換えさせる
- 差分を読まずにコミットする
- 実行していない確認をPR本文に書く

AIへの依頼例:

```txt
研修用サンプルアプリの不具合を調査しています。

再現:
未提出者フィルタを選んでも、提出済みの受講者が一覧に残ります。

観察:
- GET /api/progress?filter=unsubmitted は 200
- responseには submittedAt が入っている受講者も含まれています
- Cookieや個人情報は含めていません

出してほしいこと:
- 原因候補を3つ
- それぞれの確認方法
- まず検索すべき文字列

注意:
- 修正コードはまだ出さないでください
- 不確かなことは断定しないでください
```
