# 第20章 テクニカルライティングと知識共有 ワークブック

このワークブックは、第20章の演習「読者と文書タイプを整理する」「README更新案を書く」「設計メモを書く」「ADRを書く」「AIで文章を改善し、文書レビューを行う」で使う。

## 使い方

この資料と、第19章で作成した `ai-feature-brief.md`、`ai-diff-review.md`、`ai-work-log.md`、`ai-assisted-pr-description.md` を読み、`doc-audience-plan.md`、`readme-update-draft.md`、`feature-design-note.md`、`adr-0001-filtering-support-status.md`、`doc-review-note.md` を書く。必要に応じて `incident-report-draft.md` も作る。

目的は、文章をきれいにすることではない。実装した機能について、他の人が理解し、起動し、レビューし、運用し、将来変更できる情報を、根拠つきで残すことである。

## 前提

第19章で、次の機能を追加した想定で考える。

```txt
メンターが支援ステータス一覧を、ステータスと最終更新日で絞り込めるようにする。
```

この章では、次の文書を扱う。

- README
- 設計メモ
- ADR
- PR説明
- 手順書
- 障害報告、incident review

扱わないこと:

- ドキュメントサイト構築
- docs-as-code基盤の構築
- 広報、採用広報、マーケティング文章
- 英語ライティング専門訓練
- 法務文書の専門論点

## 演習1: 読者と文書タイプを整理する

### 考えること

文書を書き始める前に、誰が、いつ、何をするために読むかを決める。

問い:

- 新しく参加した開発者は何を知りたいか。
- レビュアーは何を確認したいか。
- 運用担当は何を知りたいか。
- 未来の自分は何を忘れていそうか。
- どの文書タイプが必要か。
- どの文書が正本か。
- 誰がいつ見直すか。
- 事実、判断、推測、未決定をどう分けるか。

### 記録すること

`doc-audience-plan.md` は次の形で書く。

```md
# 文書読者計画

## 読者

| reader | タイミング | needs | document |
| --- | --- | --- | --- |
| 新しく参加した開発者 | setup時 | 起動方法、機能概要 | README |
| レビュアー | PR review時 | 変更理由、確認結果 | PR説明、設計メモ |
| 将来の開発者 | 仕様変更時 | 判断理由、代替案 | ADR、設計メモ |
| 運用担当 | 問題発生時 | 確認手順、戻し方 | 手順書 |

## 文書タイプ
```

## 演習2: README更新案を書く

### 考えること

READMEは、リポジトリや機能の入口として、次の作業を始められる状態にする。

問い:

- この機能は何をするか。
- どう使うか。
- 起動手順や設定に変更はあるか。
- どう確認するか。
- 制約や既知の注意は何か。
- コマンドはどこで実行するか。
- 何が出れば成功か。
- READMEに入れず、別文書へリンクすべき情報は何か。

### 記録すること

`readme-update-draft.md` は次の形で書く。

```md
# README更新案

## 追加する概要

-

## 使い方

1.
2.
3.

## 動作確認

| 確認 | 実行場所 | command or operation | expected |
| --- | --- | --- | --- |
| ステータスで絞り込む | browser |    | 該当ステータスの受講者だけ表示される |
| 最終更新日で絞り込む | browser |    | 指定日以降の更新だけ表示される |
| 条件を解除する | browser |    | 一覧が通常表示に戻る |

## 設定変更

-

## 制約・注意

-

## READMEに入れない情報

-

## 関連文書

- 設計メモ:
- ADR:
- PR:

## Maintenance

- owner:
- last reviewed:
- review when:
```

## 演習4: ADRを書く

### 考えること

重要な技術判断を1つ選び、あとから判断理由を追えるようにする。

問い:

- 何の判断を残すか。
- なぜその判断が必要だったか。
- どの案を選んだか。
- どの案を選ばなかったか。
- 良い影響と悪い影響は何か。
- 中立の影響は何か。
- 将来変更するなら何を見ればよいか。

## 演習5: AIで文章を改善し、文書レビューを行う

### 考えること

AIを使って文書を読みやすくしてもよい。ただし、技術的正確性と機密情報は人間が確認する。

問い:

- AIは何を改善したか。
- 技術的な事実を勝手に追加していないか。
- 重要な判断理由を削っていないか。
- 秘密情報、個人情報、未公開情報が入っていないか。
- 読者が次の行動を取れるか。
- 事実、判断、推測、未決定が混ざっていないか。
- 実行していない確認を実行済みのように書いていないか。

### 記録すること

`doc-review-note.md` は次の形で書く。

```md
# 文書レビューメモ

## 対象文書

-

## AIに依頼したこと

-

## 採用した改善

-

## 採用しなかった改善

-

## AI推敲後の差分確認

| 観点 | 確認結果 |
| --- | --- |
| 事実の追加がない |    |
| 判断理由が削られていない |    |
| 未決定事項が断定に変わっていない |    |
| secretや個人情報がない |    |

## レビュー観点チェックリスト

- [ ] 読者が明確
- [ ] 目的が明確
- [ ] 手順が再現できる
- [ ] コマンドに実行場所と期待結果がある
- [ ] 判断理由がある
- [ ] 事実、判断、推測、未決定が分かれている
- [ ] 根拠のない追加がない
- [ ] 重要な制約や残課題が削られていない

- [ ] secretや個人情報がない

-
```

## AIを使う場合

AIに頼んでよいこと:

- 構成案
- 見出し整理
- 要約
- 読者別の書き換え
- 曖昧な表現の指摘
- 文書レビュー観点の洗い出し
- PR説明や設計メモの下書き改善

AIに任せたままにしてはいけないこと:

- 技術的事実の追加
- 判断理由の変更
- 秘密情報、個人情報、未公開情報を含む文書の貼り付け
- 障害報告の事実関係の確定
- ライセンスや法務判断
- 古い情報の更新判断
- 未実行確認を実行済みのように書くこと
- 公開リポジトリへ出せない内部URLや実データの判断

AIへの依頼例:

```txt
次の設計メモを、レビュアーが読みやすいように改善してください。

前提:
- 技術的な事実は勝手に追加しない
- 判断理由を削らない
- 制約と残課題を残す
- 秘密情報、個人情報、未公開情報は含めない

見てほしい観点:
- 読者が分かる順序になっているか
- 背景、判断、影響が分かれているか
- 曖昧な表現がないか
- PRレビューで使いやすいか
```
