# 第13章 テスト ワークブック

このワークブックは、第13章の演習「受け入れ条件からテスト観点を作る」「テストケースを書く」「TDDで小さな変更を進める」「リファクタリングメモを書く」「品質レビューを行う」で使う。

## 使い方

この資料と、第9章で作成した `acceptance-criteria.md`、第11章で作成した `api-contract.md`、第12章で作成した `ui-state-model.md`、`frontend-check-log.md` を読み、`test-strategy.md`、`test-cases.md`、`tdd-log.md`、`refactoring-note.md`、`quality-review.md` を書く。

目的は、テスト数を増やすことではない。重要なふるまいとリスクに対して、どの確認方法を選ぶかを説明し、必要なテストを書き、変更後も壊れていないことを説明できるようにすることである。

## 前提

研修用学習ログアプリに、次の機能を追加している。

```txt
メンターが、担当受講者の支援ステータスを一覧で確認し、必要に応じて変更できる。
```

これまでに作った成果物:

- 第9章: `acceptance-criteria.md`
- 第10章: `schema-draft.sql`、`sql-check-log.md`
- 第11章: `api-contract.md`、`api-check-log.md`
- 第12章: `ui-state-model.md`、`frontend-check-log.md`

この章では、これらをテスト観点とコード品質レビューに変換する。

## 演習1: 受け入れ条件からテスト観点を作る

### 考えること

受け入れ条件、API契約、画面状態を読み、確認したいリスクを整理する。

問い:

- 正常に動くことを確認する観点は何か。
- 不正な入力を確認する観点は何か。
- 権限に関する観点は何か。
- 画面状態に関する観点は何か。
- 以前壊れたら困る回帰観点は何か。
- 自動テストにするものと、手動確認に残すものは何か。

### 記録すること

`test-strategy.md` は次の形で書く。

```md
# テスト方針

## 対象機能

- 

## 守りたいふるまい

- 

## テスト観点

| category | viewpoint | リスク | 優先度 |
| --- | --- | --- | --- |
| normal |    |    |    |
| 入力検証 |    |    |    |
| 認可 |    |    |    |
| UI state |    |    |    |
| regression |    |    |    |

## 自動テストにするもの

- 

## 手動確認に残すもの

- 

## 今回対象外

```

## 演習2: テストケースを書く

### 考えること

テスト観点を、具体的な前提、操作、期待結果に変換する。

ケース候補:

- 担当メンターは担当受講者の支援ステータスを変更できる。
- 担当外メンターは変更できない。
- 不正な `status` は保存されない。
- 一覧取得に失敗したらエラーが表示される。
- 保存中は二重送信できない。
- 保存後に一覧へ反映される。

### 記録すること

`test-cases.md` は次の形で書く。

```md
# テストケース

## ケース1

名前:

- 

種類:

- unit / API / UI / manual

前提:

- 

操作:

- 

期待結果:

- 

テストデータ:

- 

確認方法:

- 

## ケース2

名前:

- 

種類:

- 

前提:

- 

操作:

- 

期待結果:

- 

テストデータ:

- 

確認方法:

- 
```

## 演習3: TDDで小さな変更を進める

### 考えること

小さなロジックを選び、red、green、refactorを記録する。

題材候補:

- `supportStatus` の値変換
- 不正な `supportStatus` の拒否
- API error codeから画面文言への変換
- 保存中フラグの扱い

### 記録すること

`tdd-log.md` は次の形で書く。

````md
# TDDログ

## 対象

- 

## 失敗（Red）

書いたテスト:

```txt

```

失敗内容:

- 

## 成功（Green）

実装したこと:

- 

通ったテスト:

- 

## 整理（Refactor）

改善したこと:

- 

変えていないふるまい:

- 

実行した確認:

- 
````

## 演習4: リファクタリングメモを書く

### 考えること

ふるまいを変えずに、コードの構造を改善する。

見る観点:

- 名前が曖昧ではないか。
- 関数やコンポーネントが長すぎないか。
- 入力検証、権限確認、DB更新、response作成が混ざっていないか。
- 同じロジックが重複していないか。
- テストしづらい依存があるか。
- コメントやTODOの意味が残っているか。

### 記録すること

`refactoring-note.md` は次の形で書く。

```md
# リファクタリングメモ

## 対象

- 

## 変えないふるまい

- 

## 見つけた問題

| 課題 | 影響 | 優先度 |
| --- | --- | --- |
|    |    |    |

## 改善方針

- 

## 実施した変更

- 

## 実行したテスト、確認

- 

## 残した負債

- 

## PRに書くこと

- 
```

## 演習5: 品質レビューを行う

### 考えること

プルリクエストに出す前に、自分の差分を読み、テストと品質の説明を作る。

問い:

- 何を変更したか。
- どの受け入れ条件を満たしたか。
- どのテストを実行したか。
- どの手動確認をしたか。
- 何を確認していないか。
- コード品質上の懸念はあるか。
- 残した技術的負債はあるか。

### 記録すること

`quality-review.md` は次の形で書く。

```md
# 品質レビュー

## 変更概要

- 

## 対応した受け入れ条件

- 

## 実行した自動テスト

- 

## 実行した手動確認

- 

## コード品質の確認

| 観点 | 結果 | メモ |
| --- | --- | --- |
| naming |    |    |
| 責務 |    |    |
| duplication |    |    |
| complexity |    |    |
| testability |    |    |

## 確認していないこと

- 

## 残したリスク、負債

- 

## レビューで見てほしいこと

- 
```

## AIを使う場合

AIに頼んでよいこと:

- テスト観点の洗い出し
- テストケースの初稿
- テスト名の改善
- テストデータの候補
- リファクタリング候補の洗い出し
- 品質レビュー観点のチェック

AIに任せたままにしてはいけないこと:

- テストの重要度判断
- 実行していないテストを通ったことにすること
- 既存テストの意図の確定
- リファクタリング後にふるまいが変わっていないことの判断
- 技術的負債を残す判断

AIへの依頼例:

```txt
支援ステータス機能のテスト観点を整理したいです。

前提:
- メンターは担当受講者だけを変更できる
- statusは none, needs_support, in_progress, resolved
- APIは 401, 403, 404, 422 を返す
- 画面には loading, empty, error, saving, saved がある
- テスト設計はプロダクトエンジニア向けの基本範囲に留める

出してほしいこと:
- 単体テストの候補
- APIテストの候補
- 画面確認の候補
- 手動確認に残すべきこと
- 見落としやすいケース
- 優先度
```

## チェックリスト

提出前に確認する。

- [ ] `test-strategy.md` に正常系、異常系、権限、状態、回帰確認の観点がある
- [ ] 自動テストにするものと手動確認に残すものが分かれている
- [ ] `test-cases.md` に前提、操作、期待結果、テストデータがある
- [ ] `tdd-log.md` にred、green、refactorがある
- [ ] `refactoring-note.md` に変えないふるまいと実行した確認がある
- [ ] `quality-review.md` にテスト結果、手動確認、未確認事項、リスクがある
- [ ] テスト名が守りたいふるまいを説明している
- [ ] 仕様変更とリファクタリングが混ざりすぎていない
- [ ] CI/CD、性能テスト、セキュリティテストの詳細に入りすぎていない
- [ ] AIを使った場合、テスト実行結果と差分で検証している
