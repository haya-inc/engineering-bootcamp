# 第4章 コラボレーション ワークブック

このワークブックは、第4章の演習「技術相談文を書く」「変更説明メモを書く」「レビューコメントに返信する」で使う。

## 使い方

この資料を読み、`technical-question.md`、`change-description.md`、`review-replies.md` を書く。目的は、きれいな文章を書くことではなく、相手が次の行動を取れる情報を渡せるようにすることである。

第4章では、GitHub上でプルリクエストを作る操作は扱わない。プルリクエストは、変更をチームにレビューしてもらうための単位として軽く紹介するだけにする。この章では、後でPR本文にも使える「変更説明メモ」を作る。

## 前提

第2章、第3章で扱ったケースを使う。

> 研修の学習ログアプリに、メンター向け進捗一覧を追加したい。

あなたは、初回リリース範囲として次を進めている。

- 担当受講者の一覧表示
- 最終ログ日時、困っていること、提出状況の表示
- 未提出者フィルタ
- CSV出力、自動アラート、詳細分析ダッシュボードは初回対象外

## 演習1: 技術相談文を書く

### 状況

未提出者フィルタを実装している途中で、次の問題が起きた。

```txt
未提出者だけを表示したいが、提出済みの受講者も一覧に残ってしまう。
```

分かっていること:

- 学習ログには `submittedAt` がある。
- 未提出の受講者は `submittedAt` が `null` になっている。
- 画面側でフィルタ処理を書いた。
- テストデータには、提出済みと未提出の受講者が混在している。
- メンター以外は進捗一覧を見られないようにする必要がある。

### 相談前に整理する

```md
## 何をしようとしているか


## 期待結果


## 実際の結果


## 確認したこと


## 判断してほしいこと


## 急ぎ度、期限、影響範囲

```

### 技術相談文の型

`technical-question.md` は次の形で書く。

```md
# 技術相談

## 状況


## 期待結果


## 実際の結果


## 確認したこと

- 
- 
- 

## 判断してほしいこと


## 補足

```

## 演習2: 変更説明メモを書く

第2章の `decision-memo.md` と第3章の `ai-use-log.md` を材料にして、変更説明メモを書く。

### 変更説明メモに含めるもの

```md
# 変更説明

## 背景


## 変更内容

- 
- 
- 

## 今回やらないこと

- 
- 

## 確認方法

- 
- 

## リスク、注意点


## レビューしてほしい点

- 
- 

## AIを使った場合の検証

```

### 悪い例

```md
進捗一覧を追加しました。
```

足りない情報:

- なぜ作ったか。
- 何を変えたか。
- 何を確認したか。
- 何をレビューしてほしいか。
- 何を初回ではやらないか。

### 良い例

```md
## 背景

メンターが詰まっている受講者に早く気づけるようにするため、進捗一覧を追加します。

## 変更内容

- 担当受講者の一覧表示
- 最終ログ日時、困っていること、提出状況の表示
- 未提出者フィルタ

## 今回やらないこと

- CSV出力
- 自動アラート
- 詳細分析ダッシュボード

## 確認方法

- 未提出者だけで絞り込めること
- メンター以外が一覧を見られないこと

## レビューしてほしい点

- 初回リリース範囲として過不足がないか
- 権限まわりの確認観点が足りているか
```

## 演習3: レビューコメントに返信する

次のレビューコメントに返信する。

### コメントA

```txt
未提出者フィルタは画面側ではなくAPI側で処理したほうがよさそうです。権限チェックとも関係しそうです。
```

返信で書くこと:

- 理解したこと
- 修正方針
- 追加する確認、テスト

### コメントB

```txt
CSV出力も今回入れたほうが便利ではないですか？
```

返信で書くこと:

- 第2章の意思決定メモとの関係
- 初回で見送る理由
- いつ再検討するか

### コメントC

```txt
困っていることの表示は、個人的な内容が含まれそうなので、閲覧範囲をもう一度確認したいです。
```

返信で書くこと:

- リスクを理解したこと
- 誰に確認するか
- 実装前に止めるべきか、仮実装で進めるか

### 返信の型

`review-replies.md` は次の形で書く。

```md
# レビュー返信

## コメントA

返信:

対応:

## コメントB

返信:

対応:

## コメントC

返信:

対応:
```

## AIを使う場合

AIに頼んでよいこと:

- 相談文の抜け漏れを確認する。
- 変更説明メモの構成を整える。
- レビュー返信の下書きを作る。
- 長いメモから要点を抜き出す。

AIに任せた後に確認すること:

- 実際に確認していないことが「確認済み」と書かれていないか。
- ログ、エラー、テスト結果が捏造されていないか。
- 第2章の `decision-memo.md` と矛盾していないか。
- 第3章の `ai-use-log.md` に、AI利用と検証結果が残っているか。

## 提出前チェック

```md
- [ ] 相談文に、状況、期待結果、実際の結果、確認したこと、判断してほしいことがある
- [ ] 変更説明メモに、背景、変更内容、確認方法、リスク、レビューしてほしい点がある
- [ ] レビュー返信に、理解、対応方針、確認事項がある
- [ ] 第2章の意思決定メモと矛盾していない
- [ ] AIを使った場合、事実と確認結果を自分で修正している
```
