# 第21章 個人開発 ワークブック

このワークブックは、第21章の演習「プロジェクトブリーフを書く」「MVPスコープを決める」「デリバリー計画を作る」「最初の縦切り実装を作る」「セルフレビューとデモ台本を書く」で使う。

## 使い方

この資料と、第1章から第20章までに作った成果物を読み、`project-brief.md`、`mvp-scope.md`、`delivery-plan.md`、`project-progress-log.md`、`self-review.md`、`demo-script.md`、`mentor-review-request.md` を作る。

目的は、大きなプロダクトを作ることではない。自由テーマを、レビュー可能で、動作確認でき、説明できる個人開発プロジェクトにすることである。

## 前提

テーマは自由とする。ただし、研修成果物として次を満たす。

- Webアプリケーションである。
- 1つ以上のユーザー操作がある。
- 1つ以上の永続化されるデータがある。
- ローカルで起動できる。
- 確認方法がREADMEに書かれている。
- 最初のデモで見せる操作が決まっている。
- 受け入れ条件と確認ログが残っている。
- 秘密情報、実在の顧客情報、実在の個人情報を扱わない。
- AIを使った場合、利用範囲と検証結果を残す。

この章では、AWSデプロイを必須にしない。デプロイや本番準備の確認は第23章で扱う。

## 演習1: プロジェクトブリーフを書く

### 考えること

自由テーマを選ぶとき、まず作りたい機能ではなく、誰の何を良くするかから考える。

問い:

- 誰が使うのか。
- その人は今、何に困っているのか。
- 作った後、何が少し良くなるのか。
- 今回扱うデータは安全に扱えるか。
- 最初のデモで何を見せるか。
- 外部API、通知、認証、決済などを使わなくても最初の価値を見せられるか。
- いちばん不確かな前提は何か。

`project-brief.md` は次の形で書く。

```md
## タイトル


## ユーザー


## 問題


## 成果


## 成功のサイン


## 基本フロー

1.
2.
3.

## 対象範囲

-

## 対象外

-

## 制約

-

## データ安全性

- 秘密情報:
- 個人情報:
- 顧客情報:
- 外部データ:

| データ | 保存するか | サンプルだけか | 削除/更新方法 | AI入力可否 |
| --- | --- | --- | --- | --- |
|    |    |    |    |    |

## 必要な証拠

-

## メンターへの質問

-
```

### チェック

- [ ] ユーザーが具体的である
- [ ] 課題が1つに絞られている
- [ ] 成果が説明できる
- [ ] 成功のサインが操作と結果で説明できる
- [ ] 扱わない範囲が書かれている
- [ ] 危険なデータを扱わない方針がある
- [ ] AIに入力してよい情報と、入力しない情報が分かれている
- [ ] いちばん不確かな前提と、確認に必要な証拠がある

## 演習2: MVPスコープを決める

### 考えること

MVPは、便利機能をたくさん入れることではない。最初に価値を確認できる最小の形を決める。

問い:

- 最初のデモで必ず必要な操作は何か。
- なくてもデモできる機能は何か。
- 今回やらないと明示した方がよい機能は何か。
- 完了したかどうかをどう確認するか。
- 成功時、失敗時または空状態、再読み込み後のどれを確認するか。

### 記録すること

`mvp-scope.md` は次の形で書く。

```md
# MVPスコープ

## プロダクト


## 必須

| 項目 | acceptance criteria | 検証 |
| --- | --- | --- |
|    |    |    |

## 推奨

-

## できれば

-

## やらない

| 項目 | 理由 |
| --- | --- |
|    |    |

## 最初のデモフロー

1.
2.
3.

## 確認する状態

| 状態 | 確認内容 |
| --- | --- |
| 成功時 |    |
| 失敗時または空状態 |    |
| 再読み込み後 |    |

## リスク

| リスク | 緩和策 |
| --- | --- |
|    |    |
```

### チェック

- [ ] Mustが3つ以内に絞られている
- [ ] Won'tが書かれている
- [ ] 受け入れ条件が確認可能である
- [ ] 成功時だけでなく、失敗時または空状態の確認がある
- [ ] 最初のデモの流れがある

## 演習3: デリバリー計画を作る

### 考えること

個人開発でも、最初から大きな実装をしない。1つのユーザー操作を、画面、API、DB、確認まで通す。

問い:

- 最初の縦切り実装は何か。
- どのデータを保存するか。
- どのAPIまたは処理が必要か。
- どの画面が必要か。
- どのテストまたは手動確認を行うか。
- AIに任せる作業はどこか。

### 記録すること

`delivery-plan.md` は次の形で書く。

```md
# デリバリー計画

## 最初の縦切り実装


## 保存するデータ

|    |    |    |    |    |

| id | task | files or area | 完了条件 | 検証 |
| --- | --- | --- | --- | --- |
| T1 |    |    |    |    |
| T2 |    |    |    |    |
| T3 |    |    |    |    |

## AIへの委任

| task | AIの役割 | human check |
| --- | --- | --- |
|    |    |    |

## 人間の確認ポイント

| タイミング | 確認すること |
| --- | --- |
| 実装前 |    |
| AI出力後 |    |
| 共有前 |    |

## コマンド

| 目的 | コマンド | 期待結果 |
```

### チェック

- [ ] 最初の縦切り実装が1つに絞られている

- [ ] AIに任せる範囲と人間の確認が分かれている
- [ ] 保存するデータの目的と安全性が説明できる
- [ ] 実装前、AI出力後、共有前、レビュー依頼前の確認がある
- [ ] 実行コマンドが書かれている

## 演習4: 最初の縦切り実装を作る

### 考えること

まず、細く動く状態を作る。画面だけ、DBだけ、APIだけを長く作り込まない。

問い:

- ローカルで起動できるか。
- 1つの主要操作が完了するか。
- データは保存されるか。
- エラー時や空状態は破綻しないか。
- 最低限の確認を実行したか。
- READMEに他者が起動できる情報があるか。

### 記録すること

`project-progress-log.md` は次の形で書く。

```md
# プロジェクト進捗ログ

## 日付: YYYY-MM-DD

### 完了

-

### 検証

| 場所 | コマンドまたは操作 | 期待結果 | 実際の結果 | result | メモ |
| --- | --- | --- | --- | --- | --- |
|    |    |    |    |    |    |

### 判断

-

### ブロッカー

-

### 次にやること

-
```

READMEには次を追記する。

```md
## ローカルセットアップ


## 実行


## 検証

| check | where | command or operation | expected | actual | result |
| --- | --- | --- | --- | --- | --- |
| setup |    |    |    |    |    |
| dev server |    |    |    |    |    |
| main flow |    |    |    |    |    |
| validation or empty state |    |    |    |    |    |
| lint/typecheck/test |    |    |    |    |    |

## 既知の制限


## デモの流れ

1.
2.
3.
```

### チェック

- [ ] ローカルで起動できる
- [ ] 1つの主要操作が端から端まで通る
- [ ] 保存後、再読み込み後、失敗時または空状態の少なくとも1つを確認している
- [ ] 確認結果がログに残っている
- [ ] READMEに起動方法と確認方法がある
- [ ] READMEの検証表に、期待結果と実際の結果がある
- [ ] 実装中の判断変更が書かれている

## 演習5: セルフレビューとデモ台本を書く

### 考えること

メンターに見せる前に、自分で成果物を確認する。完璧でなくてよいが、残課題を隠さない。

問い:

- 何の課題を解いたか説明できるか。
- 実際に動くか。
- テストまたは確認ログはあるか。
- セキュリティ上危ない入力やデータはないか。
- キーボード操作、ラベル、エラー表示など最低限のアクセシビリティを確認したか。
- READMEを見て他者が起動できるか。
- 既知の問題は何か。

### 記録すること

`self-review.md` は次の形で書く。

```md
# セルフレビュー

## 価値

- user:
- problem:
- outcome:

- demo flow:

| main flow | 合格 / 不合格 |    |
```

## AIに相談するときの例

AIに渡してよい情報だけを使い、秘密情報、実在の顧客情報、実在の個人情報は含めない。

```txt
自由テーマの個人開発プロジェクトを計画しています。

ユーザー:
研修中の自分

課題:
学習ログ、詰まったこと、次に相談したいことが散らばっています。

作りたいもの:
学習ログを登録し、状態やタグで一覧できる小さなWebアプリです。

制約:
- まずローカルで動けばよい
- 実在の個人情報は扱わない
- 最初のMVPは1週間以内に作れる範囲にしたい

お願い:
1. MVPのMust/Should/Could/Won'tを提案してください。
2. 最初の縦切り実装を1つ提案してください。
3. レビュー可能なタスクに分解してください。
4. 受け入れ条件を具体化してください。
5. 成功時、失敗時または空状態、再読み込み後の確認観点を出してください。
6. ただし、SNS化、通知、多人数共有、外部API連携、課金、複雑な認証は提案しないでください。
```

AIの出力をそのまま採用しない。自分の時間、技術力、研修の評価観点に合わせて削る。採用した提案、採用しなかった提案、検証方法を残す。
