# 第14章 セキュリティ ワークブック

このワークブックは、第14章の演習「セキュリティ観点を整理する」「認可と担当外アクセスを確認する」「ローカル演習で脆弱性を再現し、修正する」「秘密情報と依存関係を確認する」「セキュリティレビューを書く」で使う。

## 使い方

この資料と、第9章で作成した `domain-rules.md`、第11章で作成した `validation-and-authorization.md`、第13章で作成した `quality-review.md` を読み、`security-risk-note.md`、`authorization-check.md`、`attack-and-fix-log.md`、`secrets-and-dependencies-check.md`、`security-review.md` を書く。

目的は、攻撃手順を多く覚えることではない。自分の変更に対して、守るべきもの、権限、入力、出力、秘密情報、依存関係を確認し、修正と再発防止を説明できるようにすることである。

## 前提

研修用学習ログアプリに、次の機能を追加している。

```txt
メンターが、担当受講者の支援ステータスを一覧で確認し、必要に応じて変更できる。
```

この章の攻撃演習は、このリポジトリの `starter-apps/learning-log-sample` など、講師が指定したローカルの演習用アプリだけで行う。

やってよいこと:

- ローカル演習用アプリで、用意された脆弱な実装を確認する。
- ローカル環境で問題を再現する。
- 修正し、再発防止を説明する。

やってはいけないこと:

- 実在する外部サービス、社内システム、本番環境を対象にする。
- 他者のアカウントや実データを使う。
- 許可されていないスキャンや攻撃を行う。
- 秘密情報、Cookie、token、個人情報を提出物やAI入力に含める。

## 演習1: セキュリティ観点を整理する

### 考えること

問い:

- 守るべき資産は何か。
- 誰が攻撃者または誤操作の主体になりうるか。
- 起きてほしくないことは何か。

- 優先して確認すべきリスクは何か。

## 演習2: 認可と担当外アクセスを確認する

### 考えること

ログイン済みであっても、対象データを操作してよいとは限らない。ロールと担当関係で確認する。

確認する主体:

- 担当メンター
- 担当外メンター
- 未ログインユーザー
- 受講者ロール

確認する操作:

- 担当受講者一覧を見る
- 支援ステータスを更新する
- URLやrequestの `learnerId` を変更する

### 考えること

- SQLインジェクション: 文字列連結でqueryを作っている箇所。

## AIを使う場合

AIに頼んでよいこと:

- 脅威の洗い出し
- 認可確認ケースの候補
- SQLインジェクション、XSS、CSRFの基本確認観点
- 秘密情報チェックリスト
- 依存関係確認の観点
- 修正案の比較
- PRに書くセキュリティレビュー文案

AIに任せたままにしてはいけないこと:

- 秘密情報、Cookie、token、個人情報を含むログの処理
- 実在環境への攻撃手順の生成
- セキュリティ上安全であるという最終判断
- 認証認可方式の確定
- 依存関係更新の影響判断

AIへの依頼例:

```txt
支援ステータス機能のセキュリティ確認観点を整理してください。

前提:
- メンターは担当受講者だけを変更できる
- 支援メモを画面に表示する
- APIはCookieまたはAuthorization headerで認証する
- ローカル演習環境だけを対象にする
- 実在サービスへの攻撃手順は不要
- 秘密情報や個人情報は含めない

出してほしいこと:
- 守るべき資産
- 認可確認ケース
- SQLインジェクション、XSS、CSRFの基本確認
- 秘密情報とログの確認
- 依存関係の確認
- PRに書くべきセキュリティ観点
```
