# 第9章 ドメイン ワークブック

このワークブックは、第9章の演習「要件をユースケースに分解する」「用語集を書く」「状態、イベント、ルールを書く」「ドメインモデルのたたき台を書く」「受け入れ条件を書く」で使う。

## 使い方

この資料を読み、`use-case.md`、`glossary.md`、`domain-rules.md`、`domain-model.md`、`acceptance-criteria.md` を書く。

目的は、DDDの専門用語を正確に暗記することではない。実装前に、何を作るのか、どんな言葉を使うのか、どんなルールを守るのか、どこまでを今回作るのかを整理することである。

## 前提

研修用学習ログアプリに、次の機能を追加する。

```txt
メンターが、担当受講者に支援ステータスを付けられるようにしたい。
```

背景:

- メンターは、受講者の学習ログや提出状況を見て支援する。
- 受講者が増えると、誰に追加支援が必要かを一覧だけで判断しづらくなる。
- 初回リリースでは、自動アラートや詳細分析は作らない。
- メンターが手動で支援ステータスを付け、後で見返せる状態にしたい。

この章では、DB、API、画面の詳細にはまだ入らない。

## 演習1: 要件をユースケースに分解する

### 考えること

次の問いに答える。

- この機能を使う人は誰か。
- その人は何を達成したいのか。
- 事前に成り立っているべき条件は何か。
- 基本の流れは何か。
- 例外や分岐は何か。
- 成功したと言える状態は何か。
- 今回対象外にすることは何か。

### 記録すること

`use-case.md` は次の形で書く。

```md
# ユースケース

## 機能要求


## 利用者


## 目的


## 事前条件

- 

## 基本の流れ

1. 
2. 
3. 

## 例外、分岐

- 

## 成功条件

- 

## 今回対象外

- 

## 不明点

- 
```

## 演習3: 状態、イベント、ルールを書く

### 考えること

状態:

- 支援ステータスにはどんな値があるか。
- 学習ログの提出状態と支援ステータスは同じものか、別物か。
- 支援ステータスがない状態をどう扱うか。

イベント:

- 誰が何をしたときに状態が変わるか。
- 学習ログの提出で支援ステータスは変わるか。
- メンターが手動で変更するのか。

ルール:

- 誰が変更できるか。
- どの受講者について変更できるか。
- 変更履歴は必要か。
- 自動で変更される状態はあるか。

### 記録すること

`domain-rules.md` は次の形で書く。

```md

|    |    |    |
```

## 演習4: ドメインモデルのたたき台を書く

### 考えること

主要な概念と関係を整理する。

候補:

- 受講者
- メンター
- 担当
- 学習ログ
- 提出
- 支援ステータス
- 支援メモ

この章では、DBテーブルやカラム名にしない。業務上の概念として書く。

```md

- 受講者は、学習ログを提出する。
-

- 

## 重要なルール

- 

## 今回対象外の概念

```

簡単な図を書きたい場合は、次のような箇条書きでもよい。

```txt

  - 担当する -> 受講者

  - 提出する -> 学習ログ
  - 持つ -> 支援ステータス
```

## AIを使う場合

AIに頼んでよいこと:

- ユースケース案
- 用語集の候補
- 状態、イベント、ルールの抜け漏れ確認
- ドメインモデルのたたき台
- 受け入れ条件の初稿
- 不明点の洗い出し

AIに任せたままにしてはいけないこと:

- 会社やチームの業務ルールの確定
- 受講者やメンターの権限判断
- 今回対象外にする範囲の最終判断
- AIが作った用語をそのままコード名にすること

AIへの依頼例:

```txt
研修用学習ログアプリに、メンターが担当受講者へ支援ステータスを付けられる機能を追加します。

目的:
実装前に、ユースケース、用語、状態、ルール、受け入れ条件を整理したいです。

制約:
- DB設計やAPI設計にはまだ入らない
- 自動アラートは今回対象外
- 不明点は不明点として分ける

出してほしいこと:
- ユースケース案
- 用語集の候補
- 状態、イベント、ルール
- 受け入れ条件
- 確認すべき不明点
```
