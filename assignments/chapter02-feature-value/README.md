# 第2章 機能価値 ワークブック

このワークブックは、第2章の演習「小さな機能要求を分解する」と「実装方針メモを書く」で使う。

## 使い方

この資料を読み、`feature-value-breakdown.md` と `decision-memo.md` を書く。目的は、実装案を早く決めることではなく、要求を価値、制約、不明点に分けて考えることである。

## ケース

あなたは、研修で使う学習ログアプリの開発を手伝っている。

研修運営から、次の相談が来た。

> メンターが受講者の進捗を見られる一覧画面がほしいです。誰が詰まっているか分かるようにしたいです。できればCSV出力もあると助かります。

背景:

- 受講者は毎日、学習ログを書く。
- 学習ログには、今日やったこと、困っていること、次にやることが含まれる。
- メンターは複数の受講者を見ている。
- 今は、メンターが個別のログを開いて確認している。
- 研修運営は、詰まっている受講者に早く気づきたい。

## まず分ける

次の項目を埋める。

```md
## ユーザー

この機能を直接使う人:

影響を受ける人:

## 現在の課題

今、誰が何に困っているか:

## 期待する成果

この機能によって、どんな状態に変わってほしいか:

## 作るかもしれない出力

画面、API、CSV、通知、ドキュメントなど:
```

## 確認すべき不明点

実装前に確認したいことを書く。

例:

- メンターは毎日見るのか、週に数回見るのか。
- 誰がどの受講者のログを見られるべきか。
- 「詰まっている」は、どの情報から判断するのか。
- CSV出力は誰が何に使うのか。
- 最初のリリースはいつ必要か。

自分で5つ以上書く。

```md
## 確認したいこと

1.
2.
3.
4.
5.
```

## 制約を整理する

```md
## 時間の制約

いつまでに必要か:

## データの制約

どのデータが必要か:

## 権限、セキュリティの制約

誰が何を見られるべきか:

## 運用の制約

リリース後に誰がどう使い続けるか:

## 既存システムの制約

すでにある画面、DB、APIとの関係:
```

## 初回リリース範囲を決める

Must / Should / Could / Won'tで整理する。

```md
## 必須

初回リリースで必ず入れる:

## 推奨

できれば入れたい:

## できれば

余裕があれば入れたい:

## やらない

初回ではやらない:
```

記入例:

```md
## 必須

- メンターが担当受講者の一覧を見られる。
- 最終ログ日時、困っていること、提出状況を見られる。
- メンター以外は一覧を見られない。

## 推奨

- 未提出の受講者で絞り込める。

## できれば

- CSVで出力できる。

## やらない

- 自動アラート
- 詳細な分析ダッシュボード
- 受講者のスコアリング
```

## 実装案を比較する

少なくとも2案を比較する。

```md
| 案 | 内容 | 価値 | コスト | リスク | 運用負荷 |
| --- | --- | --- | --- | --- | --- |
| A |    |    |    |    |    |
| B |    |    |    |    |    |
```

例:

| 案 | 内容 | 価値 | コスト | リスク | 運用負荷 |
| --- | --- | --- | --- | --- | --- |
| A | 進捗一覧画面だけ作る | メンターが毎日確認しやすい | 中 | 権限設計が必要 | 低 |
| B | CSV出力だけ作る | 既存の表計算運用に乗せやすい | 低 | 最新状態が分かりづらい | 中 |
| C | 自動アラートまで作る | 詰まりに早く気づける | 高 | 誤通知、通知疲れ | 高 |

## 意思決定メモの型

`decision-memo.md` は次の形で書く。

```md
# 意思決定メモ

## 背景

何を相談されているか:

## ユーザーと課題

誰が何に困っているか:

## 期待する成果

この機能で何が良くなるか:

## 採用案

どの案を採用するか:

## 採用理由

なぜその案がよいか:

## 見送ったこと

今回はやらないこと:

## リスクと確認事項

注意すべきこと、実装前に確認すること:
```

## 良い例と悪い例

悪い例:

```md
一覧画面を作る。CSVも便利そうなので作る。
```

何が足りないか:

- 誰の課題か分からない。
- 成功状態が分からない。
- なぜCSVが必要か分からない。
- 権限や個人情報のリスクが見えていない。

良い例:

```md
初回は、メンターが担当受講者の最終ログ日時、困っていること、提出状況を見られる一覧画面を作る。

目的は、詰まっている受講者に早く気づけるようにすること。

CSV出力と自動アラートは初回では見送る。CSVは利用目的がまだ不明で、自動アラートは誤通知や通知疲れのリスクがあるため。

困っていることには個人的な内容が含まれる可能性があるため、閲覧権限をメンターに限定する。
```
