# 第22章 既存プロダクト改善 ワークブック

このワークブックは、第22章の演習「既存振る舞いを棚卸しする」「変更要求と影響範囲を整理する」「回帰確認計画を作る」「安全な変更計画を作る」「改善PRとリリースノートを書く」で使う。

## 使い方

この資料と、第21章で作成した `project-brief.md`、`mvp-scope.md`、`delivery-plan.md`、`project-progress-log.md`、`self-review.md`、`demo-script.md`、`mentor-review-request.md` を読み、`existing-behavior-inventory.md`、`change-impact-analysis.md`、`regression-test-plan.md`、`safe-change-plan.md`、`migration-note.md`、`improvement-pr-summary.md`、`release-note.md` を作る。

目的は、新しい機能を大きく作ることではない。すでに動いているプロダクトに、小さな改善を安全に入れることである。

## 前提

第21章で作った個人プロジェクトを、既存プロダクトとして扱う。

共通例として、次の変更要求を使ってよい。

```txt
学習ログ整理アプリにstatusを追加し、一覧でstatusによる絞り込みができるようにする。

status:
- draft
- learned
- needs-help

既存ログは learned 扱いにする。
```

自分のテーマに合わせて、同程度の小さい変更に置き換えてもよい。

変更の条件:

- 既存データが表示されなくならない。
- 既存の作成機能が壊れない。
- 不正な値を保存しない。
- READMEの起動方法と確認方法を更新する。
- 変更前の振る舞いと変更後の確認結果を、期待結果と実際の結果で残す。
- 戻す場合の注意、またはroll forwardが必要な可能性を記録する。
- 秘密情報、実在の顧客情報、実在の個人情報、本番データを扱わない。

## 演習1: 既存振る舞いを棚卸しする

### 考えること

改善を始める前に、現在の振る舞いを確認する。変更前を知らないと、変更後に何が壊れたか判断しづらい。

問い:

- 現在の主要なユーザー操作は何か。
- どの画面、API、DB、READMEが関係するか。
- 既存のテストや確認コマンドは何か。
- 既存データの形はどうなっているか。
- 変更後も守るべき振る舞いは何か。
- まだ確認できていないものは何か。

### 記録すること

`existing-behavior-inventory.md` は次の形で書く。

```md
# 既存振る舞いの棚卸し

## プロダクト


## 現在の主要フロー

| フロー | steps | 期待結果 | 実際の結果 | エビデンス | keep? |
| --- | --- | --- | --- | --- | --- |
|    |    |    |    |    | はい / いいえ |

## 現在の画面

| screen | current behavior | keep? |
| --- | --- | --- |
|    |    | はい / いいえ |

## 現在のAPIまたはサーバーアクション

| endpoint or action | current behavior | keep? |
| --- | --- | --- |
|    |    | はい / いいえ |

## 現在のデータ

| データ | フィールド | メモ |
| --- | --- | --- |
|    |    |    |

## 現在の確認

| 場所 | コマンドまたは操作 | 期待結果 | 実際の結果 | result | メモ |
| --- | --- | --- | --- | --- | --- |
|    |    |    |    |    |    |

## 守るべき振る舞い

1.
2.
3.

## まだ確認していないこと

| 項目 | 理由 | 確認方法 |
| --- | --- | --- |
|    |    |    |
```

### チェック

- [ ] 変更前の主要操作が書かれている
- [ ] 既存データの形が書かれている
- [ ] 現在の確認結果がある
- [ ] 変更後も守る振る舞いがある
- [ ] 未確認事項がある場合、理由と確認方法が書かれている

## 演習2: 変更要求と影響範囲を整理する

### 考えること

変更要求を、そのまま実装しない。目的、範囲、非対象、影響範囲に分解する。

問い:

- この変更は、誰の何を良くするのか。
- どの画面が変わるか。
- どのAPIや処理が変わるか。
- DB schemaや既存データに影響するか。
- テスト、README、手順書に影響するか。
- 互換性を壊す可能性はあるか。
- statusなどの新しい値の意味、表示名、default、既存データの扱いはそろっているか。
- validation、secret、依存関係、ログ出力への影響はあるか。

### 記録すること

`change-impact-analysis.md` は次の形で書く。

```md
# 変更影響分析

## 変更要求


## 目的


## 対象範囲

-

## 対象外

-

## 受け入れ条件

1.
2.
3.

## statusなど新しい値の契約

| value | meaning | user-facing label | created when | legacy handling |
| --- | --- | --- | --- | --- |
|    |    |    |    |    |

## validation rules

-

## 影響範囲

| 領域 | 影響 | files or evidence | リスク |
| --- | --- | --- | --- |
| UI |    |    | 低 / 中 / 高 |
| API / server action |    |    | 低 / 中 / 高 |
| DB / data |    |    | 低 / 中 / 高 |
| テスト |    |    | 低 / 中 / 高 |
| README / docs |    |    | 低 / 中 / 高 |

|    |    |    |    |    |

|    |    |
```

### チェック

- [ ] 目的と受け入れ条件がある
- [ ] 変える範囲と変えない範囲が分かれている
- [ ] UI、API、DB、テスト、READMEへの影響がある
- [ ] 新しい値の意味、表示名、default、既存データの扱いがそろっている
- [ ] 互換性リスクが書かれている
- [ ] validationやsecurityへの影響が書かれている
- [ ] 未確認事項が残っている場合、確認方法がある

## 演習3: 回帰確認計画を作る

### 考えること

改善では、新機能だけでなく、既存機能が壊れていないことを確認する。

問い:

- 変更前からある主要動作は何か。
- 新しいstatus機能で確認すべきことは何か。
- 自動テストにするものは何か。
- 手動確認で見るものは何か。
- 既存テストがない場合、現在の振る舞いを固定するテストを1つ作れるか。
- 変更前の基準と変更後の証拠をどう残すか。
- 確認しないものをどう説明するか。

### 記録すること

`regression-test-plan.md` は次の形で書く。

```md
# 回帰テスト計画

|    | 自動 / 手動 |    |    |

| --- | --- | --- |
|    |    |    |

## コマンド

| 目的 | コマンドまたは操作 | 期待結果 | 実際の結果 | result |
```

### チェック

## 演習4: 安全な変更計画を作る

### 考えること

仕様変更、リファクタリング、DB変更、文書更新を混ぜすぎないようにする。

問い:

- どの順番で変更するか。
- 先にテストを追加するか。
- リファクタリングは必要か。必要なら仕様変更と分けられるか。
- 既存データはどう扱うか。
- 戻す必要が出た場合、何を戻すか。
- AIに任せる作業はどこか。
- 各手順で、どんな状態なら止めるか。
- 新しい入力、依存関係、secret、ログ出力の影響はあるか。

### 記録すること

`safe-change-plan.md` は次の形で書く。

```md
# 安全な変更計画

## 方針


## 手順

| 順序 | 手順 | 種類 | 完了条件 | 検証 | stop if |

| 2 |    | テスト / リファクタリング / 機能 / 文書 |    |    |    |
| 3 |    | テスト / リファクタリング / 機能 / 文書 |    |    |    |

## リファクタリング境界

- needed:
- not included:
- reason:
```

## 演習5: 改善PRとリリースノートを書く

### 考えること

改善を実装したら、レビューとリリース準備につながる説明を作る。

問い:

- 何を変えたか。
- なぜ変えたか。
- 既存機能は壊れていないか。
- 既存データはどう扱ったか。
- 利用者に見える変化は何か。
- 第23章で追加確認すべきことは何か。
- 実行していない確認は何か。
- 問題が起きたらrevertかroll forwardか、どちらで考えるか。

## AIに相談するときの例

AIに渡してよい情報だけを使い、秘密情報、実在の顧客情報、実在の個人情報、本番データは含めない。

```txt
既存の学習ログ整理アプリに、小さな改善を入れます。

現在の仕様:
- 学習ログを作成できる
- ログには日付、タイトル、本文、タグがある
- 一覧でログを確認できる

変更要求:
- statusを追加する
- statusは draft / learned / needs-help
- 一覧でstatusによる絞り込みができる
- 既存ログは learned 扱いにする

制約:
- 既存ログが表示されなくならないこと
- 既存の作成機能が壊れないこと
- 不正なstatusを保存しないこと

お願い:
1. 影響しそうな領域を UI / API / DB / tests / README に分けて挙げてください。
2. 互換性リスクを挙げてください。
3. 回帰確認観点を提案してください。
4. 仕様変更とリファクタリングを分けた実装手順を提案してください。
5. 出力は仮説として書き、確認に使う検索語、見るファイル、実行するテストも提案してください。
```

AIの出力は、検索、差分、テスト、手動確認で検証する。AIが「影響なし」と言っても、関連ファイル、README、既存データ、validation、rollbackの確認を省略しない。
